# Data Room SPA — Infrastructure Setup

Цей проєкт є початковим архітектурним каркасом для Single Page Application (SPA), побудованим на стеку **Vite + React + TypeScript** із налаштованою дизайн-системою та інфраструктурними шарами.

## 🚀 Інструкція з запуску

Для локальної розробки виконайте наступні кроки:

1. **Встановлення залежностей** (якщо вони ще не встановлені):
   ```bash
   npm install
   ```

2. **Запуск сервера розробки**:
   ```bash
   npm run dev
   ```

3. **Створення збірки (Production Build)**:
   ```bash
   npm run build
   ```

## 🏗️ Опис архітектурних рішень

Структура папок організована за принципом масштабованості, модульності та чіткого розподілу відповідальності (Separation of Concerns):

*   **`src/components/ui/`** — містить атомарні UI-компоненти без бізнес-логіки (кнопки, інпути, модальні вікна тощо). Ця папка підготовлена для повної інтеграції з **Shadcn UI** на базі встановлених `tailwind-merge` та `clsx`.
*   **`src/features/`** — реалізує модульний підхід (Feature-driven architecture). Фічі є ізольованими доменами розробки:
    *   **`features/auth/`** — простір для авторизації (у майбутньому — інтеграція з Google Sign-In через `@react-oauth/google`).
    *   **`features/workspace/`** — простір для бізнес-логіки робочої області (Data Room).
*   **`src/hooks/`** — загальні кастомні React-хуки, що не прив'язані до конкретної фічі (наприклад, хуки життєвого циклу, взаємодії з DOM тощо).
*   **`src/services/`** — сервісний шар (Data Abstraction Layer). Тут будуть описані API-клієнти та Mock-сервіси, що дозволить легко перемикатися між фейковими та реальними даними без зміни UI-компонентів.
*   **`src/store/`** — глобальний стейт-менеджмент (Zustand), що ізольований від UI для кращого тестування.
*   **`src/types/`** — централізоване зберігання глобальних TypeScript типів та інтерфейсів для дотримання принципу DRY.
*   **Шляхи-аліаси (`@/*`)** — налаштовані у `tsconfig.app.json` та `vite.config.ts` для уникнення заплутаних відносних імпортів на кшталт `../../../../components`.
*   **Конфігурація середовища (`.env` / `.env.example`)** — готові змінні під майбутній деплой (`VITE_GOOGLE_CLIENT_ID`, `VITE_API_BASE_URL`).

## 📚 Інтеграція підходів з Skills.sh

У процесі налаштування інфраструктури було використано паттерни та матеріали з платформи [skills.sh](https://www.skills.sh/):

*   **Tailwind CSS & Shadcn UI Configuration**: ініціалізовано систему дизайну на основі CSS variables (класичний PostCSS-підхід + Tailwind v3), підготовлено розширення тем у `tailwind.config.js` для безшовної роботи з Shadcn UI.
*   **Production Deployment & Build Optimization**: підготовлено каркас під оптимізований SPA-деплой (розділені змінні середовища, легке підключення OAuth).
